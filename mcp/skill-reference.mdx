Each Thesis skill is a structured instruction set that guides your agent through a specific research workflow. This page describes what each of the six skill families does, when to reach for it over another, and the key behaviors your agent follows when the skill is active.

To install all skills at once, run:

```bash
npx --yes @synsci/thesis setup --install-skill
```

<AccordionGroup>
  <Accordion title="thesis / thesis-cli — general usage and platform support">
    **MCP skill:** `thesis` &nbsp;|&nbsp; **CLI skill:** `thesis-cli`

    ### Purpose

    `thesis` is the entry-point skill for platform questions, experiment design guidance, and setup or troubleshooting across all supported hosts. Use it when you want structured help planning hypotheses, choosing between node types, navigating compute options, or recovering from auth issues. Reach for a more specific skill (below) when the task is concrete graph authoring, claim validation, or autonomous execution.

    ### When to use

    - Planning experiments, hypotheses, comparisons, or next-branch decisions
    - Choosing between `insight` and `empirical` node types
    - Setting up, updating, authenticating, or troubleshooting the Thesis MCP connection
    - Questions about managed compute, campaigns, or the Thesis web UI

    ### Key behaviors

    - Classifies each request as research guidance, platform support, or mixed before acting.
    - Loads the minimum relevant reference file — one primary file, at most one supporting file — rather than reading everything upfront.
    - For research requests, asks 1–2 clarifying questions to nail objective, assumptions, and decision criteria before suggesting execution steps.
    - Uses live Thesis MCP tools to verify tool contracts when the host has MCP available.
    - Routes to specialized skills (`thesis-auto`, `thesis-reproduce`, `thesis-lookahead`, `thesis-to-graph`, `thesis-survey`) when the request calls for concrete graph work.

    ### CLI variant note

    `thesis-cli` uses the bundled bash scripts in `./scripts/` instead of `thesis_*` MCP tools. Run `./scripts/<name>.sh help` to list subcommands for any script.

    ### Install

    ```bash
    npx --yes @synsci/thesis setup --install-skill
    ```
  </Accordion>

  <Accordion title="thesis-auto / thesis-auto-cli — autonomous frontier advancement">
    **MCP skill:** `thesis-auto` &nbsp;|&nbsp; **CLI skill:** `thesis-auto-cli`

    ### Purpose

    `thesis-auto` advances a research frontier autonomously under an explicit budget. Think of it as `thesis-lookahead` plus continuous execution and replanning: the agent persists a run contract in the graph, keeps the plan `n` hops ahead, executes branches, and replans after each resolution — all within a hard budget ceiling you set before it starts.

    ### When to use

    - You want the agent to keep advancing a frontier without step-by-step user direction
    - You have a measurable objective, a budget ceiling, and an explicit terminal condition
    - You need graph-local continuation — the agent should be able to resume from node state in a fresh chat without chat history

    ### Key behaviors

    - **Requires an explicit contract before execution:** start nodes, objective, budget ceiling, budget unit, and terminal condition. The agent asks once for anything missing; if a budget remains unspecified, it plans but does not spend compute.
    - **Persists a control node:** creates a dedicated `insight` node storing the canonical run contract (`objective`, `decision criterion`, `start nodes`, `budget ceiling`, `budget unit`, `compute approval cap`, `lookahead depth n`, `frontier width k`, `terminal condition`). Later sessions read this node first instead of relying on chat memory.
    - **Keeps the graph `n` hops ahead:** after each resolved node, the agent refreshes the lookahead so staged frontier nodes always extend `n` steps beyond the current resolved edge.
    - **Executes up to `k` concurrent branches:** routes additional viable directions sequentially when more than `k` worthwhile branches exist.
    - **Uses `thesis_launch_execution`** for branches that run as Thesis node executions; uses the full compute approval flow (`thesis_request_compute_grant_approval` → `thesis_compute_acquire` → poll → connect → release) for branches that need managed hardware.
    - **Stops graph-locally:** only when the persisted terminal condition is met (objective reached, budget exhausted, or no non-redundant branch justifies remaining budget).

    ### Install

    ```bash
    npx --yes @synsci/thesis setup --install-skill
    ```
  </Accordion>

  <Accordion title="thesis-lookahead / thesis-lookahead-cli — plan the next frontier">
    **MCP skill:** `thesis-lookahead` &nbsp;|&nbsp; **CLI skill:** `thesis-lookahead-cli`

    ### Purpose

    `thesis-lookahead` stages next-step frontier nodes from existing graph state without executing them. It is planning only — no compute approval, no execution, no spend. Use it when you want to see the most promising next directions before committing to any of them.

    ### When to use

    - You want to plan what to do next without running anything yet
    - You need a staged frontier the team or another skill can review before execution
    - You want a graph-local record of considered options and the selection rationale at each step

    ### Key behaviors

    - **Single-path expansion:** at each hop, the agent evaluates up to `k` candidate directions, applies the selection criterion, and advances exactly one winner. The staged graph shape is a chain of `n` unresolved step nodes — not `k` parallel branches.
    - **Records considered options:** each step node stores the `k` options evaluated, the selection criterion used, and the chosen continuation — so the reasoning is auditable from graph state alone.
    - **Persists a planning contract:** a dedicated `insight` control node stores `objective`, `decision criterion`, `start nodes`, `lookahead depth n`, `step option width k`, and `terminal condition`. Later passes read this node first.
    - **No execution or spend:** the skill explicitly does not call `thesis_request_compute_grant_approval`, `thesis_compute_acquire`, or `thesis_launch_execution`. Execution is left to `thesis-auto` or `thesis-reproduce`.
    - **Defaults:** `n=1` (one hop ahead), `k=1` (one candidate evaluated per hop) unless you specify otherwise.

    ### Install

    ```bash
    npx --yes @synsci/thesis setup --install-skill
    ```
  </Accordion>

  <Accordion title="thesis-reproduce / thesis-reproduce-cli — graphify and validate claims">
    **MCP skill:** `thesis-reproduce` &nbsp;|&nbsp; **CLI skill:** `thesis-reproduce-cli`

    ### Purpose

    `thesis-reproduce` combines source-to-graph authoring with budgeted empirical validation. It is conceptually `thesis-to-graph` plus explicit validation branches plus compute — graphify the claim-bearing source first, split it into validation branches, then execute within a hard maximum budget you set.

    ### When to use

    - You have a claim-bearing source (paper, benchmark, experimental result) and want to validate it empirically
    - You want both the structured graph representation of the source and reproducibility branches in one workflow
    - You need a hard spend ceiling enforced before any compute runs

    ### Key behaviors

    - **Requires a hard max budget before any empirical execution:** the agent asks once if it is missing. Graph construction and branch planning may proceed without it, but execution does not start until the budget is explicit.
    - **Runs a design gate before compute:** confirms objective, decision criterion, evidence plan, branch strategy, and budget readiness using the experiment design protocol.
    - **Indexes named sources:** if the source (paper, repo, docs site) is not yet indexed, calls `thesis_index_paper`, `thesis_index_repo`, or `thesis_index_docs` before proposing validation branches.
    - **Prioritizes cheap, high-uncertainty-reduction branches first:** baseline checks, mechanism checks, ablations, efficiency checks, robustness checks, failure analysis — in that order of cost versus decision value.
    - **Persists a validation contract:** a dedicated `insight` control node stores `objective`, `decision criterion`, `source or claim nodes under test`, `budget ceiling and unit`, and `stopping criterion`. Later passes read this node first.
    - **Stops at the budget ceiling:** releases all managed compute when the core claim is resolved or the hard max budget is reached.

    ### Install

    ```bash
    npx --yes @synsci/thesis setup --install-skill
    ```
  </Accordion>

  <Accordion title="thesis-survey / thesis-survey-cli — discover and synthesize knowledge">
    **MCP skill:** `thesis-survey` &nbsp;|&nbsp; **CLI skill:** `thesis-survey-cli`

    ### Purpose

    `thesis-survey` discovers relevant external sources, cross-references prior work inside the graph, and synthesizes what is known, contested, and missing on a research question. The output is a committed `insight` node with citations — not a chat reply. Use it for literature review, research-question grounding, and identifying gaps before designing experiments.

    ### When to use

    - You want to know what is already in the graph and indexed sources on a topic before building new nodes
    - You need a structured literature synthesis with explicit citations and gap analysis
    - You are onboarding into a new area and want the agent to map the knowledge landscape first

    ### Key behaviors

    - **Graph-first retrieval:** always calls `thesis_search` with `include_graph=true` before external search. Committed graph nodes are the cheapest and most trustworthy first-pass signal.
    - **Classifies the query:** `discover` (unfamiliar question, needs sources), `validate` (claim needs supporting or contradicting evidence), `populate` (a node needs its content augmented), or `landscape` (broad area survey). Persists the classification in the node summary.
    - **Indexes named external sources before searching them:** `thesis_index_paper`, `thesis_index_repo`, or `thesis_index_docs` for any explicitly named source, so results are re-retrievable by later callers.
    - **Reads deeply before citing:** calls `thesis_read_source` on the top-3 sources for any claim that will drive a decision. Search hits are pointers, not authoritative content.
    - **Commits a structured insight node:** `summary` (one-line synopsis), `insights` (3–7 decision-relevant takeaways, non-empty or the commit is rejected), and `content` (full markdown with cited sources, plus `## Knowns`, `## Contested`, `## Gaps`, and `## Next questions` sections).
    - **No compute spend:** surveys do not acquire compute leases or request grant approval.

    ### Install

    ```bash
    npx --yes @synsci/thesis setup --install-skill
    ```
  </Accordion>

  <Accordion title="thesis-to-graph / thesis-to-graph-cli — port source material into nodes">
    **MCP skill:** `thesis-to-graph` &nbsp;|&nbsp; **CLI skill:** `thesis-to-graph-cli`

    ### Purpose

    `thesis-to-graph` converts unstructured source material — papers, READMEs, markdown wikis, research notes, blog posts — into Thesis nodes, artifacts, and durable graph edges. It is authoring only: no execution, no compute approval, no spend. Use it when the goal is to get material into the graph so agents and the web UI can navigate and search it.

    ### When to use

    - You have source material you want to turn into navigable Thesis graph nodes
    - You are onboarding a paper, repository, or wiki into Thesis before running any experiments
    - You want to give future agents (or `thesis-auto`) grounded, indexed starting material

    ### Key behaviors

    - **Classifies the input first:** if the source is already normalized Thesis graph JSON, uses `thesis_import_subgraph`; otherwise ports material explicitly with node and artifact tools (raw wikis and READMEs are not treated as direct subgraph payloads).
    - **Indexes before authoring:** calls `thesis_index_paper`, `thesis_index_repo`, or `thesis_index_docs` on named sources, then `thesis_read_source` to retrieve full text before populating node content. Cites retrieved source IDs in node content.
    - **One stable concept per node:** maps each wiki page, claim, or concept to one node. The main page markdown goes in the node `content` field; supporting files (images, PDFs, datasets, notebooks) become artifacts.
    - **Adds edges sparingly:** promotes only durable semantic relationships (hierarchy, dependency, derived-from, canonical comparison) to graph edges. Ordinary hyperlinks stay inside the node body as markdown.
    - **No execution or spend:** explicitly does not call compute approval or launch execution. Work that requires validation branches belongs to `thesis-reproduce`; autonomous continuation belongs to `thesis-auto`.
    - **Persists an authoring contract** when the source spans multiple pages or passes: a dedicated `insight` control node stores `source corpus`, `intended scope`, `root node or subtree target`, `default node granularity`, `edge policy`, and `execution policy`.

    ### Install

    ```bash
    npx --yes @synsci/thesis setup --install-skill
    ```

## Choosing between skills

<Note>
  The skills form a natural pipeline. A typical research workflow might use `thesis-survey` to map the landscape, `thesis-to-graph` to port key sources, `thesis-reproduce` to validate a central claim, `thesis-lookahead` to plan the next frontier, and `thesis-auto` to advance it under budget.
</Note>

| Goal | Skill to use |
|------|-------------|
| Platform setup, node type guidance, general Q&A | `thesis` / `thesis-cli` |
| Autonomous multi-step frontier advancement under budget | `thesis-auto` / `thesis-auto-cli` |
| Plan next steps without executing them | `thesis-lookahead` / `thesis-lookahead-cli` |
| Validate a paper or benchmark result empirically | `thesis-reproduce` / `thesis-reproduce-cli` |
| Literature review or knowledge synthesis | `thesis-survey` / `thesis-survey-cli` |
| Import source material into graph nodes | `thesis-to-graph` / `thesis-to-graph-cli` |
