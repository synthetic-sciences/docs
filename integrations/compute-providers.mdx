title: "Connect GPU compute providers: Modal and Lambda Cloud"
description: "Lease GPU instances and run sandboxed sub-agents through Modal and Lambda Cloud, with a built-in approval gate before any compute is acquired."
---

Thesis connects to two GPU compute providers. Modal and Lambda Cloud, so you can run compute-intensive research tasks without managing infrastructure yourself. Every compute action goes through an explicit approval gate: the agent proposes the work, you approve it in the UI, and only then does Thesis acquire hardware. You connect both providers from **Settings → API Keys** in the Thesis web app.

## Providers at a glance

- **Modal.** **Managed sandbox runner.** Modal provides isolated sandbox environments where Thesis sub-agents execute. When the agent spawns a sub-agent after your approval, Modal handles the container, GPU attachment, and network isolation automatically.
- **Lambda Cloud.** **GPU instance leasing.** Lambda Cloud gives you on-demand access to A10G, A100, and H100 GPU machines. Thesis acquires, monitors, and releases instances through the Lambda Cloud API on your behalf.


## Connecting a provider

Go to **Settings → API Keys** and enter your credentials for each provider you want to use:

- **Modal**: requires a Modal token ID and token secret from your [Modal dashboard](https://modal.com).
- **Lambda Cloud**: requires a Lambda Cloud API key from your [Lambda Cloud account](https://cloud.lambdalabs.com).

Once connected, the provider appears as an option when the agent proposes compute and in **Settings → Compute** where you manage active leases.

<Note>
  If a provider is not connected, it will not appear as an option during the approval flow and cannot be selected for new compute tasks.
</Note>

## Available GPU SKUs

Both providers expose the following GPU tiers in Thesis:

| SKU | GPU | Best for |
|---|---|---|
| `gpu-a10g` | NVIDIA A10G | Inference, moderate training runs, embedding workloads |
| `gpu-a100-40gb` | NVIDIA A100 (40 GB) | Large model training, high-throughput batch jobs |
| `gpu-h100` | NVIDIA H100 | Frontier model work, maximum throughput |

The agent selects a SKU based on the experiment blueprint. You can review and change the proposed SKU during the approval step before any hardware is acquired.

## The approval gate

Thesis never acquires compute without your explicit approval. The flow works as follows:

<Steps>
  <Step title="Agent proposes an experiment">
    The agent stages an experiment node with an 8-section blueprint covering the hypothesis, method, metrics, and compute requirements (provider, SKU, budget, and timeout). The proposal appears as a card in the Thesis UI.
  </Step>
  <Step title="You review and approve">
    You review the proposal in the UI. If you approve, Thesis mints an approval token and creates a compute grant. If you reject, the rejection is logged as a dead end in your research log, no compute is acquired.
  </Step>
  <Step title="Thesis acquires the lease">
    With a valid approval token, Thesis contacts the compute provider, acquires a lease, and starts the sub-agent runner. The sub-agent receives its task payload, not your chat history.
  </Step>
  <Step title="Lease is released when done">
    When the sub-agent finishes (or times out), Thesis releases the lease automatically. You can also release an active lease manually from **Settings → Compute**.
  </Step>
</Steps>

<Tip>
  Rejecting a proposal does not end the conversation. You can ask the agent to revise the experiment design, adjust the budget, or choose a different SKU before proposing again.
</Tip>

## Campaigns and budgets

A **campaign** is a named compute budget that scopes how much a series of experiments can spend. Campaigns let you authorize a block of work without approving each individual run.

### Creating a campaign

You create campaigns from **Settings → Compute → Campaigns**. Each campaign has:

- A **name** to identify the research effort.
- A **total budget** (in credits) that caps cumulative compute spending across all runs in the campaign.
- An optional **per-run limit** that caps what any single sub-agent lease can spend.

Once a campaign is active, the agent can propose experiments under that campaign. Proposals that would exceed the remaining campaign budget are blocked before they reach your approval queue.

### Monitoring a campaign

The campaign dashboard shows:

- Total budget and remaining balance.
- A list of runs completed under the campaign, each with provider, SKU, duration, and credits spent.
- Current status of any active leases associated with the campaign.

You can pause or close a campaign at any time. Closing a campaign does not terminate active leases, it prevents new leases from being opened under that budget.

## Compute leases

A **lease** represents an active or completed claim on a compute resource. Thesis tracks leases so you always know what is running and what it costs.

### Acquiring a lease

Leases are created automatically when you approve a sub-agent proposal. You can also request a manual lease from the compute panel if you need direct access to a GPU instance outside of the agent workflow.

### Checking lease status

Active leases appear in **Settings → Compute** with:

- Provider and SKU.
- Start time and elapsed duration.
- Credits consumed so far.
- Connection details (for Lambda Cloud instances: SSH host, port, and user).

### Releasing a lease

Thesis releases leases automatically when a sub-agent completes its task or reaches its timeout. To release a lease manually before it finishes:

1. Open **Settings → Compute**.
2. Find the active lease in the list.
3. Click **Release**.

Manual release immediately terminates the compute session and stops billing for that lease.

<Note>
  Released leases cannot be resumed. If you release a lease while a sub-agent is still running, the agent's task will fail. Make sure the agent has finished or saved its work before releasing manually.
</Note>

## What happens without a compute provider

If neither Modal nor Lambda Cloud is connected:

- The agent can still propose experiments and you can approve them, but the sub-agent runner cannot start.
- No hardware is acquired or billed.
- Thesis logs the provider-not-configured state so you can diagnose it in the research log.

You do not need both providers. Connect only the one that fits your workflow, or connect both and let the agent select based on the experiment requirements.
