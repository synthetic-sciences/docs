Synsci CLI ships with 83 machine-learning skills bundled into the binary. Each skill is a self-contained instruction set the agent loads on demand — when your prompt matches a skill's trigger, the relevant playbook joins the context, the agent follows it, and unloads when done. No external install step, no PATH wiring.

## What a skill is

A skill is a markdown playbook plus optional helper scripts that teach the agent how to do one thing well. For ML work, skills cover:

- **Training frameworks** — PyTorch DDP, DeepSpeed, FSDP, Axolotl, Megatron
- **RLHF and post-training** — GRPO, PPO, DPO, RLOO, KTO
- **Inference and serving** — vLLM, TensorRT-LLM, SGLang, llama.cpp
- **Evaluation** — lm-eval-harness, BIG-bench, HELM
- **Fine-tuning** — LoRA, QLoRA, ReFT, full-parameter SFT
- **Distributed and infra** — Ray, SkyPilot, Slurm, K8s job templates
- **Model and data hubs** — Hugging Face datasets and models, datasets streaming
- **Experiment tracking** — Weights & Biases, MLflow

Each skill is short and focused: how to start a job, what flags to pass, common pitfalls, where outputs land, and how to read the failure modes.

<Note>
  The full list is exposed in the CLI itself. Run `synsc` to start a session, then ask: "List the bundled skills you have access to." The agent enumerates them from its context.
</Note>

## How skills load

Skills are not always-on. The agent loads a skill into context only when your prompt is on-topic for that skill. This keeps the model's context lean and lets the binary ship 83 skills without making every session pay for all of them upfront.

A typical flow:

1. You ask: *"Set up a GRPO training run on the smollm-1.7B base with the dolma corpus."*
2. The agent matches the prompt against the bundled skills, finds the GRPO playbook, and loads it.
3. The skill tells the agent how to scaffold the config, what trainer flags matter, where to point at the dataset, and what to log.
4. The agent executes — writing files, kicking off the run, streaming logs.
5. Once the task is done or you change topic, the skill unloads.

You don't manage the lifecycle. The agent handles it.

## Trigger phrases

You don't need to invoke skills by name. Just describe the task in normal language and the agent picks the right skill:

| If you ask… | The agent loads… |
|---|---|
| "Fine-tune llama-3-8b with QLoRA on this dataset" | LoRA + QLoRA + transformers training skill |
| "Spin up vLLM with tensor parallel 2 for this model" | vLLM serving skill |
| "Run GRPO with a reward model and log to wandb" | GRPO + W&B skill |
| "Submit this script to a 4xH100 node on Modal" | Modal job skill |
| "Evaluate this model against MMLU" | lm-eval-harness skill |

## Skills and credentials together

Skills know about the credentials Synsci CLI syncs from your dashboard. When the GRPO skill needs to log to Weights & Biases, it uses the W&B token already in your environment — no manual setup. When the Modal skill spins up a sandbox, it uses your Modal token. When `from datasets import load_dataset` needs Hugging Face auth, your HF token is there.

See [Credentials](/cli/credentials) for which tokens sync and how they reach subprocesses.

## When a bundled skill isn't enough

If you need a workflow that isn't covered by a bundled skill:

- **Describe it in detail** in your prompt. The model is capable enough to scaffold most ML jobs from a careful description.
- **Add it to the skills repo.** The Synsci skills library is open and accepts contributions. The maintained library lives at [github.com/SyntheticSciences/AI-research-SKILLs](https://github.com/SyntheticSciences/AI-research-SKILLs).
- **Use a custom skill in your project.** Drop a `.synsc/skills/<name>.md` file in your repo and the agent will load it alongside the bundled skills for sessions started in that directory.
