title: "Build a first graph"
description: "Create a root node, branch the work, attach evidence, and render the result."

---

A map is useful when each step says what changed and why. Start with one root question, then add small child nodes for plans, experiments, results, and summaries.

## Create the root

```bash
atlas nodes:create \
  --payload-json '{
    "local_temp_node_id": "tmp-baseline-question",
    "parent_ids": [],
    "staged_payload": {
      "title": "evaluate sparse attention baseline",
      "content": "Which baseline should we trust before running ablations?",
      "summary": "Root question for the baseline evaluation.",
      "kind": "question"
    }
  }' \
  --format=json
```

Copy the returned `node_id`. The examples below use `nd_root`.

## Add a plan branch

```bash
atlas nodes:branch \
  --node-id nd_root \
  --title "baseline plan" \
  --format=json
```

Use a branch when the child work depends on the parent but may diverge. If you need a fully shaped plan node, use `nodes:create` with `parent_ids`.

## Commit a result

```bash
atlas nodes:create \
  --payload-json '{
    "local_temp_node_id": "tmp-baseline-result",
    "parent_ids": ["nd_plan"],
    "staged_payload": {
      "title": "baseline result",
      "content": "The baseline trains successfully; validation loss plateaus after epoch 18.",
      "summary": "Baseline run completed and produced usable metrics.",
      "kind": "result"
    }
  }' \
  --format=json
```

Keep result nodes factual. Put interpretation in a separate claim or summary node so later work can challenge it cleanly.

## Attach evidence

```bash
atlas evidence:add \
  --node-id nd_result \
  --items '[{
    "local_path": "./runs/baseline/metrics.json",
    "artifact_type": "json",
    "note": "raw baseline metrics"
  }]' \
  --format=json
```

Evidence should be the raw material a reviewer needs to verify the node. If the support already lives in an indexed library source, link the source path in the node summary instead of uploading a duplicate.

## Render the tree

```bash
atlas map:tree --node-id nd_root --max-depth 4
atlas map:summary --node-id nd_root --format=json
```

Use `map:tree` for a terminal view. Use `map:view` when you need a structured projection for a script or UI.

## Good graph hygiene

- Prefer small nodes over long mixed notes.
- Use links to preserve why a node exists.
- Upload raw files as evidence; save interpretation as nodes.
- Use labels for status like `planned`, `running`, `validated`, or `blocked`.
- Use `--format=json` when an agent is reading the output.
