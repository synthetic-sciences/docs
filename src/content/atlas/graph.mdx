Atlas stores research state as committed nodes connected by links. Each node should answer one question: what changed in the research state?

If you are new to Atlas, read [Build a first graph](/first-graph) before using this reference.

## Concepts

| Concept | Meaning |
| --- | --- |
| Node | A durable research object with title, kind, summary, metadata, evidence, and revision history. |
| Save | A recorded state change. Save changes agents or humans should be able to audit later. |
| Link | A directed relationship from prior context to derived work. Branching and merging are map operations, not folders. |
| Access policy | The visibility and access policy for one node or a batch of nodes. |
| Draft lock | A short-lived edit lock for multi-agent work on the same node. |
| Label | A root-scoped marker for organizing map state and tracking status. |
| Linked evidence | A public HTTPS JSON record resolved into a node without uploading a local file. |
| Run | A node-scoped record for started or stopped work. |

## Pick the right node shape

| Work | Suggested kind | Summary should contain |
| --- | --- | --- |
| Starting point | `question` | The question and why it matters. |
| Proposed work | `plan` | The next action and success condition. |
| Run output | `result` | The observed result, not a broad interpretation. |
| Interpretation | `claim` | The claim and the evidence that supports it. |
| Source note | `observation` | The library-backed fact and citation path. |
| Review state | `summary` | What is known, unknown, and next. |

## Minimal flow

```bash
atlas nodes:create \
  --payload-json '{
    "local_temp_node_id": "tmp-sparse-attention-question",
    "parent_ids": [],
    "staged_payload": {
      "title": "test sparse attention baseline",
      "content": "Which sparse attention baseline should we trust?",
      "summary": "Question node for baseline selection.",
      "kind": "question"
    }
  }' \
  --format=json

atlas nodes:branch --node-id nd_... --title "run reference baseline"
atlas map:tree --node-id nd_...
```

Use `nodes:create` when creating durable state. `nodes:branch` is a convenience branch operation with title-level metadata. Use `draft:create` only when you need a lower-level staged flow.

## Node lifecycle

| Command | Use |
| --- | --- |
| `draft:create` | Create an uncommitted node shell. |
| `nodes:create` | Create and save a new node. |
| `nodes:save` | Save staged changes to an existing node. |
| `nodes:show` | Inspect one node. |
| `nodes:list` | List visible nodes. |
| `nodes:files` | List files attached to one node. |
| `nodes:delete` | Delete one node. |
| `nodes:delete-bulk` | Delete multiple nodes. |

`nodes:create` accepts the full commit-new envelope through `payload_json`. Agents should inspect `atlas help nodes:create --schema --format=json` before building payloads.

## Links and topology

| Command | Use |
| --- | --- |
| `map:view` | Read a map projection with depth, paging, and filters. |
| `nodes:branch` | Create a child branch. |
| `nodes:merge` | Merge nodes into a resolved node. |
| `link:add` | Add a dependency link. |
| `link:remove` | Remove a dependency link. |
| `link:parents` | Page direct parents. |
| `link:children` | Page direct children. |
| `nodes:resolve` | Resolve a slug to a node. |
| `program:snapshot` | Resolve a root snapshot for a node. |

Use `nodes:merge` only when there is a resolved synthesis. Do not merge unresolved alternatives just to shorten the tree.

## Rendered map views

```bash
atlas map:lineage --node-id nd_... --max-depth 4 --format=json
atlas map:summary --node-id nd_... --max-nodes 50 --format=json
atlas map:tree --node-id nd_... --max-depth 5
```

Use rendered views for terminal agents. Use `map:view` when you need the raw projection for your own UI or script.

## Sharing

```bash
atlas access:show --node-id nd_... --format=json
atlas access:set --node-id nd_... --visibility public --format=json
atlas access:summaries --node-ids nd_1,nd_2 --format=json
```

| Command | Use |
| --- | --- |
| `access:show` | Read one node policy. |
| `access:set` | Change visibility. |
| `access:set-bulk` | Set visibility or access on many nodes. |
| `access:summaries` | Read policy summaries for many nodes. |

## Labels

```bash
atlas labels:create --root-node-id nd_... --name validated --format=json
atlas labels:assign --node-id nd_... --tag-ids tag_... --format=json
```

| Command | Use |
| --- | --- |
| `labels:create` | Create a reusable label. |
| `labels:assign` | Assign labels to nodes. |
| `labels:update` | Change label text, colors, or behavior. |
| `labels:delete` | Delete a label. |

## Concurrency

Use draft locks when multiple agents may edit the same node:

```bash
atlas draft:lock --node-id nd_...
atlas draft:renew --node-id nd_... --stage-session-id session_...
atlas draft:unlock --node-id nd_... --stage-session-id session_...
```

The draft lock commands take a `stage_session_id`. A long-running agent should lock the draft, renew it while editing, and unlock it with a reason when the staged work is committed or abandoned.

## Linked evidence

Linked evidence records are public HTTPS JSON records attached to a node by URL.

```bash
atlas evidence:link \
  --node-id nd_... \
  --remote-artifact-id metrics-live \
  --url https://example.com/results.json \
  --title "external run metrics" \
  --rows-path data.rows \
  --row-id id \
  --format=json

atlas evidence:refresh \
  --node-id nd_... \
  --remote-artifact-id metrics-live \
  --url https://example.com/results.json \
  --title "external run metrics" \
  --rows-path data.rows \
  --row-id id \
  --format=json
```

## Runs

Runs are node-scoped work records. They are useful when an agent starts, lists, or stops work tied to a map node.

```bash
atlas runs:start --node-id nd_... --provider local --format=json
atlas runs:list --node-id nd_... --format=json
atlas runs:stop --node-id nd_... --execution-id ex_... --yes
```
