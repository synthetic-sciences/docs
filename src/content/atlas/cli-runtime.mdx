The Atlas CLI is a direct HTTP runtime plus a few local commands. It does not hide a second command surface behind the docs.

## Config resolution

Atlas resolves runtime config in this order:

| Setting | Resolution order |
| --- | --- |
| API key | `--api-key`, `ATLAS_API_KEY`, profile `api_key_env`, profile `api_key`, root `api_key` |
| Base URL | `--base-url`, `ATLAS_BASE_URL`, profile `base_url`, root `base_url`, production `/api/v1` |
| Profile | `--env`, `ATLAS_PROFILE`, active config profile, `default` |
| Config path | `ATLAS_CLI_CONFIG_PATH`, then `~/.config/atlas-cli/config.json` |

`atlas login` writes a profile node with `0600` file permissions. `atlas logout` removes the stored credentials for the active profile.

## Global flags

| Flag | Use |
| --- | --- |
| `--format <format>` | Output as `json`, `tsv`, `csv`, `table`, or `pretty`. |
| `--env <profile>` | Select a CLI profile. |
| `--out <path>` | Write output to a file, or `-` for stdout. |
| `--open` | Open browser-bound URLs for commands that support it. |
| `--no-retry` | Disable transient-error retries. |
| `--retry-max <n>` | Set retry attempts. |
| `--timeout <seconds>` | Set request timeout. |
| `--idempotency-key <key>` | Pass a stable key for retried writes. |
| `--yes` | Confirm destructive operations. |
| `--force` | Force safety-gated operations where supported. |
| `--debug` | Print structured request debugging. |

## Help and schemas

Use help before automating a command:

```bash
atlas help --format=json
atlas help nodes:create --schema --format=json
```

The help payload comes from the same command registry that dispatches requests. It includes command names, groups, endpoints, required fields, optional fields, and JSON schema.

## Local and HTTP commands

`login`, `logout`, and `doctor` are local commands. They can call Atlas APIs, but they also manage local state.

Every other command maps to one of the concrete routes shown in [Commands](/commands). For example:

```text
atlas map:view -> GET /api/v1/graph
atlas library:grep -> POST /api/v1/sources/{source_id}/grep
atlas map:summary:pdf -> POST /api/v1/export-summary-pdf
```

## Upload handling

`evidence:add` is the only command that performs a multi-step local workflow:

1. Validate local upload items.
2. Prepare upload records with Atlas.
3. PUT file bytes to the signed upload URL.
4. Finalize the batch.

The runtime allows up to `50` upload items and `100 MB` per file. Supported artifact types include `text`, `table`, `json`, `image`, `banner`, `html`, `plotly_html`, `vega`, `checkpoint`, `binary`, and `diff_carousel`.

Use lower-level `evidence:prepare` and `evidence:finalize` only when you need custom upload control.

## Raw, binary, and streaming output

Some commands do not return normal JSON:

| Command | Output shape |
| --- | --- |
| `history:export` | Raw JSONL. |
| `map:summary:pdf` | Binary PDF. |
| `map:summary:render-pdf` | Binary PDF. |
| `map:summary:stream` | Streaming response. |
| `library:runs:stream` | Server-sent job events. |
| `research:runs:stream` | Server-sent job events. |

Use `--out <path>` for binary exports.

## Request echo

Set `ATLAS_HTTP_ECHO=1` to inspect request shaping without issuing the real request for supported local/upload paths.

```bash
ATLAS_HTTP_ECHO=1 atlas login --format=json
```

Use this for debugging wrappers and agent integrations.
