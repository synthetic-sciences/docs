You need Node `18+` and access to an Atlas account. The CLI uses browser login, then stores an Atlas API key in your local config. Atlas uses direct REST and CLI commands; there is no public Atlas MCP server.

<Steps>
  <Step title="Install the CLI">
    ```bash
    npm i -g @synsci/atlas@latest
    atlas --version
    ```
  </Step>

  <Step title="Sign in">
    ```bash
    atlas login
    ```

    The CLI opens a browser approval flow, redeems a one-time token, and stores a local Atlas API key.
  </Step>

  <Step title="Verify the runtime">
    ```bash
    atlas doctor --format=json
    atlas whoami --format=json
    atlas help --format=json
    ```

    `doctor` checks config, auth, backend reachability, and bundled skill presence.
  </Step>

  <Step title="Create a node">
    ```bash
    atlas nodes:create \
      --payload-json '{
        "local_temp_node_id": "tmp-first-question",
        "parent_ids": [],
        "staged_payload": {
          "title": "first research question",
          "content": "What should we verify first?",
          "summary": "Initial Atlas research question.",
          "kind": "question"
        }
      }' \
      --format=json
    ```

    Save the returned `node_id`. Most map, evidence, and library workflows start from a node.
  </Step>
</Steps>

## What success looks like

`atlas doctor --format=json` should report a reachable backend, an authenticated profile, and bundled skills. `atlas whoami --format=json` should return the signed-in account. `atlas help --format=json` should return the command registry.

If login succeeds but commands fail, run doctor before changing config. It is the fastest way to separate a local config issue from an API issue.

## Learn the surface

After setup, run:

```bash
atlas help --format=json
atlas help map:view --format=json
atlas help nodes:create --schema --format=json
```

`help --format=json` is generated from the live command registry. Use it when writing scripts or agent prompts because it tells you required flags, optional flags, endpoint mapping, and schema.

## Local configuration

Atlas reads credentials in this order:

1. `--api-key`
2. `ATLAS_API_KEY`
3. the active profile written by `atlas login`
4. root config fallback

Useful environment variables:

| Name | Purpose |
| --- | --- |
| `ATLAS_BASE_URL` | Override the default API base URL. |
| `ATLAS_CLI_CONFIG_PATH` | Override the local config file path. |
| `ATLAS_PROFILE` | Select the active CLI profile. |
| `ATLAS_HTTP_ECHO` | Print request shaping for local debugging. |
