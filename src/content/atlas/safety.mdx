Atlas is designed for human and agent collaboration. Automation should be explicit about state changes.

The CLI enforces one important rule locally: commands marked destructive in the registry require `--yes` or `--force`.

## Destructive commands

Delete, revoke, remove, terminate, and cancel commands should be run deliberately. Use `--yes` only after checking the target IDs.

Examples:

```bash
atlas nodes:delete --node-id nd_... --yes
atlas evidence:delete --node-id nd_... --artifact-id art_... --yes
atlas keys:delete --key-id key_... --yes
atlas library:runs:cancel --job-id job_... --yes
```

Destructive families include node deletion, evidence deletion, library source deletion, run cancellation, GitHub disconnect, integration removal, key deletion, and key rotation.

## Agent output

Prefer machine-readable output:

```bash
atlas help --format=json
atlas map:view --format=json
atlas library:search --query "..." --format=json
```

When retrying writes from an external orchestrator, pass an explicit idempotency key:

```bash
atlas nodes:create \
  --payload-json '{
    "local_temp_node_id": "tmp-retry-safe-node",
    "parent_ids": [],
    "staged_payload": {
      "title": "retry-safe node",
      "content": "This node was created by a retry-safe wrapper.",
      "summary": "Retry-safe node creation example."
    }
  }' \
  --idempotency-key "agent-run-123-node-1" \
  --format=json
```

Use `expected_revision` on map and evidence writes when the command supports it. That makes stale writes fail instead of silently overwriting newer state.
