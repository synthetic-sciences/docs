Synci runs locally with the same filesystem and shell access you do. The security model is built around two principles: keep provider credentials out of subprocesses that do not need them, and make the trust boundary explicit instead of pretending the CLI is a sandbox.

This page describes what the CLI does for you and what is still your responsibility.

## What the CLI is not

**The CLI is not a sandbox.** The permission and approval system around destructive tool calls is a UX guardrail to keep you in the loop, not isolation. The agent can read, write, and execute anywhere your user can. If you need real isolation, run Synci inside Docker, a VM, or a remote dev container.

## Credential storage

| Surface | Where | Notes |
| --- | --- | --- |
| Synci Bearer token | `$XDG_CONFIG_HOME/synsc/config.json` (defaults to `~/.config/synsc/config.json`) | `thk_<uuid>.<secret>`, 1-year TTL, revocable from the dashboard. |
| Synced service credentials | `$XDG_CONFIG_HOME/synsc/credentials.json` | Encrypted at rest. Refreshed by `synsci connect sync`. |
| Native binary (curl-bash install) | `~/.synsc/bin/synsc` | Added to PATH through your shell rc. |

The dashboard holds the canonical copy of every synced credential. The local file is a session-time mirror that you can delete and re-pull with `synsci connect sync`.

## Subprocess env hygiene

When the agent shells out to run a tool, the CLI rebuilds the subprocess environment from a curated allow-list rather than inheriting your full shell environment. This is what keeps provider API keys from leaking into commands that have no business seeing them.

**Always passed through (the `SAFE_ENV_PREFIXES` list):** `PATH`, `HOME`, `USER`, `SHELL`, `TERM`, `LANG`, `LC_*`, `TMPDIR`, `XDG_*`, `EDITOR`, `VISUAL`.

**Conditionally passed through, only to subprocesses that declare a need:** `HF_TOKEN`, `WANDB_API_KEY`, `MODAL_TOKEN_ID`, `MODAL_TOKEN_SECRET`, `LAMBDA_API_KEY`, `RUNPOD_API_KEY`, `PRIME_INTELLECT_API_KEY`, `TENSORPOOL_API_KEY`, `VAST_API_KEY`, `TINKER_API_KEY`, `LANGSMITH_API_KEY`, `PINECONE_API_KEY`, `TOGETHER_API_KEY`, `GROQ_API_KEY`, `FIREWORKS_API_KEY`, `OPENROUTER_API_KEY`.

**Explicitly filtered out (the `SHARED_PROVIDER_KEYS` list), even if they exist in your shell:** `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `GOOGLE_GENERATIVE_AI_API_KEY`, `GEMINI_API_KEY`. The agent talks to model providers itself; subprocesses do not need these keys, and stripping them prevents accidental exfiltration.

The full lists live in the source at [`src/synsci/index.ts`](https://github.com/synthetic-sciences/synsci-cli/blob/main/backend/cli/src/synsci/index.ts).

## Output redaction

Known credential patterns are redacted from agent output before it lands in the transcript or syncs back to the dashboard. The redaction matches the synced-key shape (`hf_*`, `wandb-*`, Modal token IDs, AWS access keys, etc.). If you spot an unredacted secret in a transcript, file an issue against [synthetic-sciences/synsci-cli](https://github.com/synthetic-sciences/synsci-cli/issues) — the pattern set is updated as new providers ship.

## Server mode

`synsci serve` exposes the agent over HTTP for headless or remote use:

```bash
synsci serve --port 4444
```

By default, server mode logs a warning and binds without authentication. Set `SYNSCI_SERVER_PASSWORD` to require HTTP Basic Auth on every request before exposing the port beyond `127.0.0.1`. Never expose `synsci serve` directly to the public internet without an upstream proxy that enforces auth, rate limiting, and TLS.

## Per-account isolation

Sessions and synced credentials are scoped to your Synthetic Sciences account. The dashboard never mixes one account's state with another's. Team plans add explicit project-level sharing; defaults are private.

Each device approval is independent. Revoking a device from the dashboard invalidates only that device's Bearer token; other devices on the same account keep working.

## What leaves your machine

- Prompts and assistant responses you send during a session (stored against your dashboard account).
- Synci credit-usage telemetry per session, when you are on the managed route.
- Model provider traffic, when you use a managed or hosted provider (each provider has its own data policy).

## What stays on your machine

- Source files the agent reads or writes locally, unless you explicitly upload them through a tool that takes a file as an argument.
- Local environment variables that are not in the allow-list above.
- The native binary, the local config directory, and any session metadata stored under `~/.config/synsc`.

## Reporting a vulnerability

Security reports go through GitHub Security Advisories on the [synsci-cli repo](https://github.com/synthetic-sciences/synsci-cli/security/advisories) with `security@anoma.ly` as a fallback. Do not post security issues on public issue trackers.
