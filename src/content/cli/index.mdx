Synci is a terminal coding agent purpose-built for ML engineering and scientific research workflows. The published binary is `synsci`. It pairs with the Synthetic Sciences dashboard through a shared account, but most of the agent runs locally on your machine.

The CLI bundles a terminal agent, an optional local web UI, model routing across hosted and BYOK providers, a credentials sync that pulls keys for HF, W&B, Modal, Lambda, RunPod, and friends, and a library of research skills you can pull into any session. The dashboard at [app.syntheticsciences.ai/cli](https://app.syntheticsciences.ai/cli) handles billing, key management, and device approval.

## How Synci works

Synci's job is to be the agent loop you actually run on your laptop: keystrokes, files, processes, model calls, and credentials. The dashboard's job is to be the account: provisioned keys, billing, sync, and the device-approval endpoint that bootstraps the local install.

The product vocabulary is small:

- A **session** is a single agent conversation with its own message history, tool calls, and working directory. Sessions are local first and optionally synced.
- A **provider** is a model backend that the agent can route through: Anthropic, OpenAI, OpenAI Codex (sign-in-with-ChatGPT), Google, OpenRouter, Together, Fireworks, Groq, and the managed Synthetic Sciences route.
- An **agent profile** is a system prompt plus a tool set plus a routing policy. The built-in `synsci` profile is the default; you can create sub-agents with `synsci agent create`.
- A **skill** is a portable instruction bundle that primes the agent for a domain. The CLI ships an ML-focused skill library covering training, tokenization, fine-tuning, evaluation, RAG, agents, distributed compute, and more.
- A **synced credential** is a third-party API key or token issued by you in the dashboard and pulled into the local CLI process at runtime, never written to disk under your project tree.

The usual loop is: install the binary, run `synsci connect login` once, optionally sign into any external providers you use, then run `synsci` to start a session, `synsci run "..."` for one-shot work, or `synsci web` for a local browser UI.

## How a command works

Every CLI invocation routes through a small set of local concerns: argument parsing, model and provider selection, credential sourcing, tool gating, and output formatting. Anything that needs network goes to either the model provider you picked or to the Synthetic Sciences backend at `https://thesis-synsc.fly.dev` for account, billing, sync, and device-approval calls.

The loop is:

```text
agent or human
  -> synsci command
  -> local config in ~/.config/synsc
  -> provider route (Anthropic/OpenAI/Codex/...) or Synci backend
  -> tool calls (bash, files, MCP, skills, ...) executed locally
  -> JSON, stream, file, table, or pretty output
```

`connect`, `auth`, `models`, `billing`, `mcp`, `skill`, and `agent` all read or write local CLI config; they call the Synci backend when they need to. Everything else maps to a model call plus local tool work.

## Minimum working loop

```bash
npm i -g synsci
synsci connect login
synsci connect status

synsci "Refactor utils/log.ts to use structured logging"
```

After that, use `synsci run` for one-shot prompts, `synsci web` for the local browser UI, `synsci skill add` to load a research skill, and `synsci models` to see what providers are configured.

## Boundary

Synci is not a sandbox. The permission and approval system is a UX guardrail to keep you in the loop on destructive tool calls; it is not isolation. If you need isolation, run Synci inside Docker or a VM. Synced credentials are redacted from terminal output and stripped from subprocess environments by default, but the agent has the same filesystem and shell access you do.
