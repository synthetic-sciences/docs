A **skill** is a portable instruction bundle the agent loads into the active session to prime it for a domain. Skills are not modes the CLI runs in; they are pluggable knowledge that travels with a session and changes how the agent thinks about a task. You can add the built-in skills, install third-party skills from a URL, or pin a set of skills to a project.

The Synci CLI ships with a research-skills library — 82 skills across 20 categories — covering the ML-engineering surface most teams hit: model architecture, tokenization, fine-tuning, RLHF and post-training, mechanistic interpretability, distributed training, inference serving, evaluation, MLOps, multimodal, RAG, agents, and more. The full skill set lives at [github.com/synthetic-sciences/synsci-cli/tree/main/research](https://github.com/synthetic-sciences/synsci-cli/tree/main/research).

## Skill commands

| Command | Use |
| --- | --- |
| `synsci skill list` | Show installed skills, both built-in and user-added. |
| `synsci skill show <name>` | Print one skill's metadata and entrypoint instructions. |
| `synsci skill add <url-or-name>` | Install a skill from a URL (Git or HTTPS bundle) or by name from the built-in library. |
| `synsci skill remove <name>` | Remove an installed skill. |
| `synsci skill set-entries <name>` | Update the skill's entrypoint mapping (which tools it injects, when it activates). |

Run `synsci skill list` to see every skill on your install. Skills are versioned, so a project can pin to a specific revision.

## Add a third-party skill

Anyone can publish a skill as a Git repository or HTTPS bundle. To install:

```bash
synsci skill add https://github.com/example-org/my-skill
synsci skill list
```

The CLI fetches the bundle, validates its manifest, and registers it under the name declared in the manifest. After install, the skill is available in any future session.

## How a skill activates

A skill ships an entry manifest that declares: a name, a description, an activation rule (always-on, project-scoped, or invoked by name), the tools it wants the agent to be able to call, and the system-prompt fragment it contributes. At session start, the agent loads every always-on and project-scoped skill that matches the working directory, plus any skill you invoke by name during the session.

Activation is additive. Loading the `vllm-serving` skill does not unload the `distributed-training` skill; the agent gets both contexts.

## Pin skills to a project

Create `.synsc/skills.json` in your project root:

```json
{
  "skills": [
    { "name": "vllm-serving", "version": "1.x" },
    { "name": "evaluation-harness", "version": "1.x" }
  ]
}
```

Every session started in that directory loads exactly those skills. CI agents and team members get the same primed agent without coordinating one-off install commands.
